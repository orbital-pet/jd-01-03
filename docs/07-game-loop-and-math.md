# 07. 게임 루프와 게임 수학 — 게임이 "살아 움직이는" 원리 ❤️

> 이 문서가 이 시리즈의 심장입니다. 여기 나오는 개념들(dt, 지수 감쇠,
> 거리 충돌, 상태 기계, 수명 기반 이펙트)은 어떤 게임을 만들든,
> 아니 게임이 아니어도 애니메이션이 있는 모든 UI에서 평생 쓰입니다.

---

## 1. 게임 루프 — 전부 이것의 반복입니다

`joops-game.tsx`의 `loop` 함수:

```ts
const loop = (now: number) => {
  const dt = Math.min(0.05, (now - last) / 1000);  // ① 흐른 시간 계산
  last = now;
  t += dt;                                          // ② 누적 시간
  update(dt);                                       // ③ 세상을 조금 움직이고
  draw();                                           // ④ 그 순간을 사진 찍는다
  raf = requestAnimationFrame(loop);                // ⑤ 다음 프레임 예약
};
```

게임은 **"시간을 조금 흘려보내고(update) → 그 순간을 그린다(draw)"**를
1초에 60번 반복하는 것뿐입니다. 이게 전부예요. 진짜로.

### update와 draw를 왜 나누나

- `update`: 세상을 **바꾸기만** 한다. 캔버스를 절대 만지지 않는다.
- `draw`: 세상을 **읽기만** 한다. 아무것도 바꾸지 않는다.

이 규칙 덕분에 디버깅이 절반으로 줄어요:
"숫자가 이상하다 → update만 본다 / 화면이 이상하다 → draw만 본다."

## 2. `dt` (델타 타임) — 게임 프로그래밍의 알파이자 오메가

`requestAnimationFrame`의 호출 간격은 기기마다 다릅니다:

- 60Hz 모니터 → 약 16.7ms마다
- 120Hz 아이패드 → 약 8.3ms마다
- 렉 걸리면 → 40ms, 100ms…

만약 이렇게 짜면:

```ts
j.y += 2;   // ❌ "한 프레임에 2픽셀" — 120Hz 폰에선 두 배 빨리 떨어짐!
```

게임이 기기마다 다른 게임이 됩니다. 그래서 속도를 **"1초에 몇 픽셀"**로
정의하고, 실제로 흐른 시간을 곱합니다:

```ts
j.y += j.vy * dt;   // ✅ vy는 "초당 픽셀", dt는 "흐른 초"
```

60Hz에서는 작은 걸음을 60번, 120Hz에서는 더 작은 걸음을 120번 —
**결과적으로 1초에 같은 거리**를 갑니다. 이것이 dt의 존재 이유예요.

### `Math.min(0.05, ...)` — dt 클램프

탭을 백그라운드에 30초 뒀다 돌아오면 dt가 30(초)이 됩니다. 그러면
쓰레기가 한 걸음에 수천 픽셀을 순간이동해서 주인공을 **통과**해 버려요
(터널링이라고 부릅니다). 충돌 판정이 통째로 증발하죠.

상한을 0.05초로 자르면, 아주 느린 프레임에서는 게임이 슬로모션이 될지언정
물리는 절대 깨지지 않습니다. **정확성보다 안정성을 택한 것.**

### `t` — 누적 시간, 모든 "살아있음"의 재료

`t`는 게임 시작 후 흐른 초. `Math.sin(t * 속도)`는 -1↔1을 오가는 값이라
어디에 곱하든 흔들림이 됩니다:

| 코드 | 효과 | 어디 |
|---|---|---|
| `Math.sin(t * 2) * 3` | 안테나가 좌우로 3px 하늘하늘 | `mascot.ts` |
| `Math.sin(t * 5 + seed)` | 별의 두근두근 맥박 | `debris.ts` |
| `Math.floor(t * 16) % 2` | 초당 8번 켜졌다 꺼졌다 = 무적 깜빡임 | `joops-game.tsx` |
| `0.5 + 0.5 * Math.sin(t * 2)` | 달의 "z" 잠꼬대가 숨쉬듯 밝아졌다 어두워졌다 | `backdrop.ts` |

📌 **`+ seed` 트릭**: 모두 같은 `t`를 쓰면 모든 물체가 동시에 똑같이
움직여서 기계 같아 보입니다. 물체마다 다른 상수(seed)를 sin 안에 더하면
위상이 어긋나 제각각 살아 움직여요. 모션 그래픽의 단골 기법입니다.

## 3. sin/cos 30초 완성

수학 시간에 배운 삼각함수, 게임에서는 딱 두 용도입니다:

**용도 ①: 왔다갔다 (진동)** — `sin(시간 × 빠르기) × 폭`

```ts
j.x = j.x0 + Math.sin(j.swayT) * j.swayAmp;   // debris.ts — 좌우 하늘하늘
```

**용도 ②: 각도 → 좌표 (원 그리기)** — `x = cos(각도) × 반지름, y = sin(각도) × 반지름`

```ts
// doodle.ts — 원 둘레에 점 12개 찍기
const a = (i / n) * Math.PI * 2;
pts.push([x + Math.cos(a) * rr, y + Math.sin(a) * rr]);

// effects.ts — 무작위 방향으로 입자 발사
vx: Math.cos(a) * sp,  vy: Math.sin(a) * sp,
```

이 두 패턴만 알면 이 게임의 삼각함수는 전부 읽힙니다.

## 4. 지수 감쇠(Exponential Smoothing) — 부드러운 움직임의 정체 ⭐

주인공이 손가락을 딱딱 따라오지 않고 **살짝 늦게 미끄러지듯** 따라오는
손맛의 8할이 이 세 줄입니다:

```ts
const k = Math.min(1, dt * 7);
player.x += (player.tx - player.x) * k;   // tx = 목표(손가락) 위치
player.y += (player.ty - player.y) * k;
```

읽는 법: **"목표까지 남은 거리의 일부(k)만큼, 매 프레임 다가간다."**

```
프레임 1: 남은 거리 100 → 12만큼 이동 → 남은 거리 88
프레임 2: 남은 거리  88 → 10만큼 이동 → 남은 거리 78
프레임 3: 남은 거리  78 →  9만큼 이동 → …
→ 가까워질수록 이동량이 줄어든다 = 자연스러운 감속 (ease-out)
```

`* 7`이 **반응 속도**입니다. 이 숫자만 바꿔 보세요 (`TUNE.followSpeed`):

| 값 | 느낌 |
|---|---|
| `dt * 2` | 굼벵이. 젤리 속을 헤엄치는 느낌 |
| `dt * 7` | 현재 값. 쫀득하게 따라옴 |
| `dt * 30` | 거의 즉시. 손가락에 딱 붙음 |

`Math.min(1, ...)`이 필요한 이유: 렉으로 dt가 커지면 `dt * 7 > 1`이 되어
목표를 **지나쳐서 진동**합니다. 1로 잘라 오버슈트를 막아요.

이 패턴은 이 게임 곳곳에서 재사용됩니다 — 입 벌리기(`dt * 8`),
눈동자 시선(`dt * 6`), 쓰레기 빨려들기(`dt * 20`), 자석(`dt * 3`).
**한 줄짜리 관용구로 외워 두면 평생 씁니다:**

```ts
현재값 += (목표값 - 현재값) * Math.min(1, dt * 속도);
```

<details>
<summary>🎓 심화: 사실 이건 근사입니다</summary>

엄밀히는 `dt`가 달라지면 결과가 미묘하게 달라집니다. 수학적으로 정확한
프레임 독립 버전은 `k = 1 - Math.exp(-속도 * dt)`예요. 다만 dt가 작을 때
`1 - e^(-s·dt) ≈ s·dt`라 현재 코드는 좋은 근사이고, dt가 0.05로 클램프되어
있어 오차가 눈에 띄지 않습니다. 그래서 대부분의 게임이 그냥 곱셈을 씁니다.
리플레이 저장이나 네트워크 동기화처럼 정밀함이 필요해지면 exp 버전을 쓰세요.
</details>

## 5. 충돌 판정 — 물리 엔진 필요 없어요

원과 원의 충돌은 피타고라스 정리 한 줄입니다:

```ts
const dist = Math.hypot(j.x - player.x, j.y - player.y);  // 두 중심의 거리
if (dist < player.r + j.size) { /* 겹쳤다! */ }
```

```
     ●←──── dist ────→●
    반지름A          반지름B        dist < A + B  ⇔  두 원이 겹침
```

### 히트박스는 보이는 것과 다르다 — 실무의 핵심 노하우

`joops-game.tsx`의 update를 보면 판정 크기가 그림과 다릅니다:

```ts
// 가시(위험물): 그림은 가시 끝까지 size * 1.5인데, 판정은 훨씬 작게
if (player.invul <= 0 && dist < player.r + j.size * 0.75) ...

// 먹을 것: 입을 벌린 만큼(+ mouth * 10) 판정이 커짐
if (dist < player.r + j.size * 0.65 + player.mouth * 10) ...
```

> 🎮 **황금률: 플레이어에게 유리한 쪽으로 거짓말하라.**
> 피격 판정은 보이는 것보다 작게 (가시가 스쳐도 안 죽음 → 억울함 제거),
> 획득 판정은 보이는 것보다 크게 (잘 먹힘 → 기분 좋음).
> 마리오도 젤다도 슈팅게임도 전부 이렇게 합니다.

### 자석 — 모바일이라 필요한 배려

폰 화면에서 손가락으로 픽셀 단위 조준은 불가능합니다. 그래서 70px 안에
들어온 먹이는 게임이 슬쩍 끌어당겨 줘요 (`TUNE.magnetRange`).
플레이어는 "내가 잘 먹었네!"라고 느끼지만 사실 게임이 도와준 겁니다.

```ts
if (dist < player.r + TUNE.magnetRange) {
  const pull = Math.min(1, dt * 3);
  j.x0 += (player.x - j.x) * pull;   // ⚠️ x가 아니라 x0!
  j.y  += (player.y - j.y) * pull;
}
```

**⚠️ 반드시 짚고 갈 함정**: 왜 `j.x`가 아니라 `j.x0`를 옮길까요?
`stepDebris`를 보면 `j.x = j.x0 + Math.sin(...) * 폭` — 즉 `x`는 매 프레임
`x0`(흔들림의 중심축)로부터 **다시 계산되는 파생값**입니다. `x`를 고쳐봤자
다음 프레임에 덮어써져요. 진짜로 옮기려면 원본인 `x0`를 옮겨야 합니다.

📌 일반 원칙: **파생된 값을 고치지 말고, 그 값의 원본을 고쳐라.**
(반면 `y`는 `y += vy * dt`로 누적되는 원본이라 직접 고쳐도 됩니다.)

### 무적 시간 (i-frame)

```ts
player.invul = TUNE.invulTime;   // 맞으면 1.4초 무적
```

이게 없으면 가시 하나에 몸이 걸쳐 있는 동안 **매 프레임** 맞아서
하트 3개가 0.05초 만에 증발합니다. 피격 후 무적은 선택이 아니라 필수.
무적 동안 주인공을 깜빡이게 해서(draw의 `globalAlpha = 0.4`)
플레이어에게 상태를 알려주는 것까지가 한 세트입니다.

## 6. 상태 기계(FSM) — 게임의 교통정리

```ts
type Phase = "title" | "playing" | "over";
let phase: Phase = "title";
```

게임 전체가 이 변수 하나로 정리됩니다:

```
     ┌───────────────── start() ──────────────────┐
     │                                             │
  title ── 탭 ──▶ playing ── 하트 0 ──▶ over ── 탭(0.6초 후) ─┘
```

`phase`는 코드 곳곳에서 스위치 역할을 해요:

- `if (phase !== "playing") continue;` → 타이틀/게임오버에서는 충돌 판정 없음
  (타이틀 화면의 쓰레기가 주인공을 그냥 통과하는 건 **의도된** 동작)
- 타이틀에서는 가시가 안 나오고, 낙하 속도도 60%로 느긋하게 (spawn 참고)

📌 게임오버 직후 0.6초간 입력 무시(`if (t - overAt < 0.6) return;`)는
죽는 순간 누르고 있던 손가락 때문에 결과 화면을 읽기도 전에 재시작되는
사고를 막는 장치입니다. 이런 걸 **입력 디바운스**라고 해요.

## 7. 스폰과 난이도 곡선 — 재미를 설계하기

### 카운트다운 스폰

```ts
spawnIn -= dt;
if (spawnIn <= 0) {
  spawn();
  const base = phase === "playing" ? Math.max(0.42, 1.05 - score / 900) : 1.5;
  spawnIn = base * (0.7 + Math.random() * 0.6);   // ±30% 흔들기
}
```

- `1.05 - score / 900`: 점수가 오를수록 간격이 줄어 어려워짐.
- `Math.max(0.42, ...)`: 바닥을 둬서 **무한히 어려워지지 않게** 함.
- `* (0.7 + random * 0.6)`: 완벽히 일정한 리듬은 외워져서 지루합니다.
  **불규칙성도 의도적으로 설계하는 것.**

### 확률 구간 나누기

```ts
const difficulty = Math.min(1, score / 400);   // 0 → 1로 서서히
const roll = Math.random();
if (roll < 0.07) kind = "star";                                   // 7%
else if (phase === "playing" && roll < 0.17 + difficulty * 0.13) kind = "hazard"; // 10%→23%
else kind = pickEdible();                                         // 나머지
```

`[0, 1)` 난수를 자로 재듯 구간으로 자르는 패턴:

```
0        0.07              0.17~0.30                            1.0
├─ 별 7% ─┼─ 가시 10%→23% (난이도에 따라 벽이 오른쪽으로) ─┼─ 일반 쓰레기 ─┤
```

난이도가 오르면 가시의 구간이 넓어지며 일반 쓰레기의 몫을 잡아먹습니다.
**게임 밸런스가 코드 세 줄**인 거예요. 이 숫자들을 바꿔 보세요!

## 8. 수명 기반 이펙트 — age / life 만능 공식

`effects.ts`의 팝업과 스파크는 같은 설계를 씁니다:

```ts
s.age += dt;                                     // 나이를 먹고
if (s.age >= s.life) sparks.splice(i, 1);        // 수명이 다하면 제거
ctx.globalAlpha = 1 - s.age / s.life;            // 진행도만큼 흐려짐
```

`age / life`는 **0에서 1로 가는 정규화된 진행도**입니다. 투명도, 크기,
위치, 색… 어디에든 곱하면 애니메이션이 됩니다.

응용 — 팝업의 "지연 페이드":

```ts
ctx.globalAlpha = f < 0.7 ? 1 : 1 - (f - 0.7) / 0.3;
// 처음 70%는 완전 불투명 → 마지막 30%에서만 사라짐
// "충분히 읽게 해준 다음에 사라진다" — 텍스트 피드백의 기본기
```

`eatT`도 비슷한 아이디어입니다. `-1`이면 "해당 없음", `0~1`이면 먹히는
진행도 — **숫자 하나로 상태와 진행도를 함께** 표현하는 요령이에요
(`-1` 같은 특별한 값을 센티널이라고 부릅니다).

## 9. 게임 필(feel)을 만드는 잔기술 모음

같은 로직이라도 이런 디테일이 "손맛"을 만듭니다. 이 게임에서 찾아보세요:

| 기법 | 코드 위치 | 효과 |
|---|---|---|
| 화면 흔들기 | `draw()`의 shake | 맞았을 때의 타격감 |
| 여러 겹 이펙트 | `swallow()` — 소리+진동+입자+팝업+커짐 | 먹는 맛! 하나만 빼도 밋밋해짐 |
| 빨려들며 회전+축소 | `stepSwallow` + `drawDebris`의 scale | "꿀꺽" 느낌 |
| 입 벌리기 예고 | update의 `wantOpen` | 먹기 전에 기대감 조성 |
| 시선 추적 | `player.look` | 캐릭터가 "생각하는 것처럼" 보임 |
| 커지는 몸 | `TUNE.growPerEat` | 성장의 즐거움 + 리스크 증가 |
| 별의 하트 회복 | `swallow()`의 `lives < 3` 분기 | 위기 탈출의 드라마 |
