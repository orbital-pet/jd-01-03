# 01. 프로젝트 한 바퀴 투어 🚌

> 이 문서를 읽고 나면: 프로젝트를 실행할 수 있고, 어떤 파일이 무슨 일을 하는지 알고,
> 게임이 돌아가는 전체 그림이 머릿속에 그려집니다.

---

## 1. 이 프로젝트는 무엇인가요?

**「SPACE JOOPS · 우주 냠냠!」** — 손가락(또는 마우스)으로 입 큰 친구를 움직여서
하늘에서 떨어지는 우주쓰레기를 먹어치우는 웹 게임입니다.

- 그림은 전부 **코드로 실시간으로 그립니다.** 이미지 파일이 한 장도 없어요.
- 소리도 전부 **코드로 실시간 합성합니다.** mp3 파일도 없어요.
- 사용한 기술: **Next.js 16 + React 19 + TypeScript + Tailwind CSS 4 + Canvas 2D**

"게임은 게임 엔진으로 만드는 것 아닌가요?" — 아니에요! 웹 브라우저 자체가
훌륭한 게임기입니다. 이 프로젝트가 그 증거예요.

---

## 2. 실행해 보기

터미널에서 프로젝트 폴더로 이동한 뒤:

```bash
# 처음 한 번만: 필요한 라이브러리 설치 (node_modules 폴더가 생깁니다)
npm install

# 개발 서버 켜기
npm run dev
```

그리고 브라우저에서 **http://localhost:3000** 을 여세요.

| 명령어 | 하는 일 |
|---|---|
| `npm run dev` | **개발 서버.** 코드를 저장하면 화면이 즉시 바뀝니다(핫 리로드). 평소엔 이거! |
| `npm run build` | **제품용 빌드.** 배포할 수 있는 최적화된 결과물을 만들고, 타입 오류도 잡아줍니다. |
| `npm run start` | 빌드한 결과물로 실제 서비스처럼 서버를 켭니다. |
| `npm run lint` | 코드 스타일/실수 검사기(ESLint)를 돌립니다. |

> 💡 **꿀팁**: `npm run dev`를 켜 놓고 코드 숫자를 바꿔 저장해 보세요.
> 게임이 실시간으로 변합니다. 이 "만지작거리기"가 최고의 공부법이에요.

## 3. 폴더 구조 — 누가 어디 사는지

```
jd-01/
├── app/                  ← ★ 우리가 만드는 모든 것 (아래에서 자세히)
├── docs/                 ← 지금 읽고 있는 학습 문서들
├── public/               ← 그대로 서빙되는 정적 파일 (지금은 svg 아이콘뿐)
├── node_modules/         ← npm install이 설치한 라이브러리들 (직접 수정 금지!)
├── package.json          ← 프로젝트의 신분증: 이름, 의존 라이브러리, 명령어 목록
├── package-lock.json     ← 설치된 라이브러리의 정확한 버전 기록 (자동 관리)
├── tsconfig.json         ← TypeScript 설정
├── next.config.ts        ← Next.js 설정 (지금은 기본값)
├── postcss.config.mjs    ← Tailwind CSS를 동작시키는 설정
└── eslint.config.mjs     ← 코드 검사기 설정
```

### app/ 폴더 — 진짜 우리 코드

Next.js에서 `app/` 폴더는 특별합니다. **폴더 구조가 곧 웹사이트 주소**가 돼요.

```
app/
├── page.tsx              ← 주소 "/"      = 첫 화면
├── play/
│   └── page.tsx          ← 주소 "/play"  = 게임 화면
├── layout.tsx            ← 모든 페이지를 감싸는 공통 껍데기
└── globals.css           ← 사이트 전체 스타일
```

규칙은 딱 두 개입니다:

1. **폴더 이름 = 주소.** `app/play/` 폴더 → `/play` 주소.
2. **그 폴더의 `page.tsx` = 그 주소에서 보여줄 화면.**

나머지 파일(`joops-game.tsx`, `lib/*.ts` 등)은 주소가 되지 않습니다.
`page.tsx`가 아니니까요. 그냥 페이지들이 가져다 쓰는 부품이에요.

### 부품들의 관계도

```
        (첫 화면 /)                       (게임 화면 /play)
        app/page.tsx                      app/play/page.tsx
         │       │                              │
         ▼       ▼                              ▼
  doodle-sky  best-score              app/play/joops-game.tsx ★게임 본체
  (배경 데모)  (최고기록)                       │
         │       │                              ▼
         │       │                    app/play/game-ui.tsx (점수판·타이틀·게임오버)
         │       │                              │
         └───────┴──────────┬───────────────────┘
                            ▼
                       app/lib/ ← 공용 부품 창고
      ┌──────────┬──────────┬──────────┬──────────┬─────────┐
   doodle.ts  debris.ts  mascot.ts  backdrop.ts  effects.ts  …
   (손그림 펜)  (쓰레기)    (주인공)     (배경)      (이펙트)
```

핵심 관찰 두 가지:

- **첫 화면과 게임 화면이 `lib/`의 같은 부품을 공유합니다.**
  첫 화면 배경에서 혼자 노는 마스코트와, 게임에서 조종하는 마스코트는
  똑같은 `drawMascot` 함수로 그려져요. 부품을 잘 나누면 이런 재사용이 공짜입니다.
- **`lib/` 파일들은 React를 전혀 모릅니다.** 평범한 TypeScript 함수와 타입뿐이에요.
  "그림 그리기와 게임 규칙"을 "화면 관리(React)"와 분리해 둔 것 — 이 프로젝트에서
  제일 중요한 설계입니다.

## 4. 게임이 돌아가는 큰 그림 — 세 겹의 세계

`/play` 화면은 세 겹으로 이루어져 있습니다. 이걸 구분하는 게 첫 번째 관문이에요.

```
┌────────────────────────────────────────────────────────┐
│ ① Next.js 서버 컴포넌트   app/play/page.tsx              │
│    · 탭 제목, 모바일 화면 설정(metadata/viewport)만 담당    │
│    · 브라우저에서 실행되는 JS 없음. HTML만 만들어 보냄       │
└──────────────────────┬─────────────────────────────────┘
                       │ <JoopsGame />을 화면에 올림
┌──────────────────────▼─────────────────────────────────┐
│ ② React 클라이언트 컴포넌트   JoopsGame()                 │
│    · 점수판/타이틀/게임오버 같은 "글자 UI" 담당             │
│    · 초당 60번이 아니라 "점수가 바뀔 때만" 다시 그림         │
│    · React가 아는 상태는 useState의 ui 꾸러미 하나뿐        │
└──────────────────────▲─────────────────────────────────┘
                       │ pushUi()로만 소통 (한 방향!)
┌──────────────────────┴─────────────────────────────────┐
│ ③ 게임 세계   useEffect 안의 클로저                       │
│    · 초당 60번 도는 게임 루프 (update → draw → 반복)       │
│    · 상태는 전부 평범한 지역 변수 (let score, let junks…)   │
│    · React를 전혀 모름. <canvas>에 직접 그림               │
└────────────────────────────────────────────────────────┘
```

### 왜 이렇게 나눴을까? — 이 코드의 가장 중요한 설계 결정

게임은 1초에 60번 화면을 다시 그립니다. 만약 게임 상태(쓰레기 위치, 점수…)를
React의 `useState`에 넣으면, 1초에 60번 React 전체가 다시 렌더링됩니다.
폰에서는 그 비용을 감당하지 못해 게임이 뚝뚝 끊겨요.

그래서 이 코드의 원칙은:

> **초당 60번 변하는 것은 캔버스에 직접 그린다.**
> **가끔 변하는 것(점수, 하트)만 React에게 알린다.**

게임 세계가 React에게 말을 거는 유일한 통로가 `joops-game.tsx`의 `pushUi()`
함수입니다. 쓰레기를 먹거나, 맞거나, 게임이 시작/종료될 때 — 즉 점수판이
실제로 바뀌어야 할 때만 호출돼요. 초당 60번이 아니라 초당 2~3번입니다.

📌 이 패턴은 게임만이 아니라 지도, 차트, 그림판처럼 "그림은 빠르게, UI는 가끔"
갱신해야 하는 모든 웹앱에 그대로 쓰입니다. 기억해 둘 가치가 충분해요.

## 5. 코드를 읽는 추천 순서

막히지 않고 읽으려면 이 순서를 따라가 보세요:

1. **`app/lib/constants.ts`** — 제일 쉬움. 색과 종류 정의를 눈에 익히기.
2. **`app/play/joops-game.tsx`의 `loop` 함수** — 게임의 심장. 딱 8줄입니다.
3. **같은 파일의 `update` 함수** — 게임 규칙 전부가 여기 있어요.
4. **같은 파일의 `draw` 함수** — 그리는 순서(배경 → 쓰레기 → 주인공 → 이펙트).
5. **`app/lib/debris.ts`** — "데이터 + 움직이는 함수 + 그리는 함수" 3종 세트의 교과서.
6. **`app/lib/doodle.ts`** — 손그림의 비밀. [08 문서](./08-doodle-rendering.md)와 함께.
7. **`app/play/game-ui.tsx`와 `app/page.tsx`** — React/JSX 연습.
8. **`app/components/doodle-sky.tsx`** — 본체를 이해했다면 복습 겸 술술 읽힙니다.

## 6. 자주 나오는 질문

**Q. 이미지 파일이 없는데 그림은 어디서 나와요?**
전부 `<canvas>`에 코드로 그립니다. `lib/mascot.ts`를 열면 주인공의 눈, 입,
볼터치를 원과 곡선으로 그리는 코드가 있어요. [06 문서](./06-canvas-2d.md) 참고.

**Q. 게임 상태가 `useState`에 없는데 어떻게 동작해요?**
`useEffect(() => { ... }, [])` 안의 지역 변수들은 **클로저**라는 JS의 성질 덕분에
게임이 끝날 때까지 살아 있습니다. [02 문서](./02-javascript-typescript.md)의
클로저 절을 보세요.

**Q. `.ts`와 `.tsx`의 차이는?**
`.tsx`는 TypeScript + JSX(HTML처럼 생긴 문법). 화면 조각(컴포넌트)을 만드는
파일이 `.tsx`, 순수 로직만 있는 파일이 `.ts`입니다. 이 프로젝트에서
`lib/`는 전부 `.ts`, 화면 부품은 전부 `.tsx`인 것도 그래서예요.

**Q. 코드를 잘못 만져서 망가뜨렸어요!**
괜찮아요, 그러라고 있는 게 git입니다. `git status`로 뭘 바꿨는지 보고,
`git checkout -- 파일명`으로 그 파일만 원상 복구할 수 있어요.
마음껏 부수면서 배우세요. 진짜로요.
