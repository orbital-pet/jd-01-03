## 1. 목표: 손그림처럼 보이려면 무엇이 필요한가

손으로 그린 애니메이션을 관찰하면 두 가지 특징이 있습니다:

1. **선이 삐뚤빼뚤하다** — 자로 그은 직선이 없다.
2. **선이 미묘하게 끓는다(boiling)** — 같은 그림을 여러 장 그려서 넘기면
   장마다 선이 조금씩 달라서, 정지 그림인데도 살아 있는 것처럼 일렁인다.

①은 쉬워 보입니다. 선의 점들을 난수로 흔들면 되니까요. 그런데 함정이 있어요.

## 2. 왜 `Math.random()`으로는 안 되는가 🔑

매 프레임 이렇게 그린다고 해봅시다:

```ts
// ❌ 이렇게 하면 안 됩니다
ctx.lineTo(x + (Math.random() - 0.5) * 3, y + (Math.random() - 0.5) * 3);
```

`Math.random()`은 부를 때마다 다른 값을 줍니다. 초당 60번 다시 그리는
캔버스에서는 **매 프레임 흔들림이 완전히 새로 뽑혀서**, 그림이 형체를
못 유지하고 부들부들 떨게 됩니다. (직접 해보세요. 눈이 아픕니다.)

필요한 것은 **"난수처럼 보이지만, 같은 질문에는 항상 같은 답을 주는"** 함수입니다.
이걸 결정론적(deterministic) 난수라고 해요.

### `wob()` — 결정론적 가짜 난수

```ts
// doodle.ts
const wob = (seed: number, i: number, ph: number = phase) => {
  const v = Math.sin(seed * 127.1 + i * 311.7 + ph * 74.7) * 43758.5453;
  return (v - Math.floor(v)) * 2 - 1;   // 결과: -1 ~ +1
};
```

한 줄씩 해부하면:

1. `Math.sin(...)` — 입력들을 섞어서 -1~1 사이의 매끄러운 값으로.
2. `* 43758.5453` — 엄청나게 큰 수로 뻥튀기. 입력이 조금만 달라져도 결과가 확 달라짐.
3. `v - Math.floor(v)` — **소수 부분만 남김** (12345.678 → 0.678).
   큰 수의 소수 부분은 사실상 예측 불가 → 난수처럼 보임.
4. `* 2 - 1` — 0~1 범위를 -1~+1로 변환.

이상한 마법 상수들(127.1, 311.7, 43758.5453)은 셰이더(GPU 그래픽) 세계의
유명한 관용구입니다. GLSL의 `fract(sin(dot(...)) * 43758.5453)`을 본 적이
있다면 같은 물건이에요.

**핵심 성질**: `wob(3, 5, 7)`은 언제 어디서 불러도 항상 같은 값입니다.
그래서 같은 (seed, i, phase) 조합으로 그리는 한, 매 프레임 다시 그려도
**똑같은 삐뚤빼뚤함** = 형체가 유지됩니다.

세 개의 입력은 각각:

| 입력 | 의미 |
|---|---|
| `seed` | **어느 물체**의 흔들림인지 (물체마다 고유 번호) |
| `i` | 그 물체의 **몇 번째 점**인지 |
| `ph` | 지금이 **몇 번째 장면**인지 (↓ 3절) |

## 3. "끓는 선"의 비밀 — phase의 계단식 증가

`joops-game.tsx`의 `draw()` 첫 줄:

```ts
d.setPhase(Math.floor(t * 7));
```

`t`(누적 시간)는 매끄럽게 증가하지만, `Math.floor(t * 7)`은
**초당 7번만 계단식으로 바뀌는 정수**입니다:

```
t (연속):   0.00  0.05  0.10  0.15  0.20  0.25  0.30
phase:        0     0     0     1     1     1     2
            └── 같은 그림 ──┘└── 다른 그림 ──┘└─ 또 다른…
```

- phase가 같은 동안(약 0.14초 = 8~9프레임)은 **완전히 똑같은** 삐뚤빼뚤함.
- phase가 바뀌는 순간 모든 선이 한꺼번에 새로운 모양으로 톡 바뀜.

이게 바로 손그림 애니메이션의 **boiling line** 효과입니다. 연필로 같은
그림을 7장 그려서 1초에 걸쳐 넘겨 보는 것과 같아요.

**실험해 보세요** (이 시리즈에서 제일 재밌는 실습입니다):

| `d.setPhase(...)`를 이렇게 바꾸면 | 결과 |
|---|---|
| `d.setPhase(0)` | 삐뚤빼뚤하지만 **정지**. 종이에 그린 정물화 |
| `Math.floor(t * 3)` | 느긋하게 꿈틀. 나른한 느낌 |
| `Math.floor(t * 7)` | **현재 값.** 활기찬 낙서 애니메이션 |
| `Math.floor(t * 30)` | 지지직. 노이즈처럼 눈이 아픔 |
| `t` (floor 없이) | 매끄럽게 일렁임 — 손그림이 아니라 **물속 해파리**처럼 보임 |

마지막 줄이 핵심 통찰입니다. **`Math.floor`(양자화)가 없으면 손그림이
아닙니다.** 디지털적인 "딱딱 끊김"이 오히려 아날로그 느낌을 만드는 역설이에요.

## 4. `wobblyPath` — 직선을 손그림 선으로

```ts
const wobblyPath = (pts: Pt[], seed: number, amp: number, close: boolean) => { ... }
```

꼭짓점들을 받아서 ① 각 변을 2등분해 점을 늘리고, ② 모든 점을 `wob`으로
`amp`만큼 흔들고, ③ 흔들린 점들을 직선으로 잇습니다.

**왜 점을 늘리나?** 꼭짓점만 흔들면 변 자체는 여전히 자로 그은 직선입니다:

```
점 안 늘림:  A ~~~ B          꼭짓점만 삐뚤, 변은 곧음 → 어색
점 늘림:     A ~·~ B          변의 중간도 휘어짐 → 손으로 그은 선!
```

코드에서 `for (let s = 0; s < 2; s++)`가 그 세분화입니다. 2를 3, 4로
늘리면 더 구불구불해져요. (직접 바꿔 보세요!)

`close` 인자: `true`면 마지막 점→첫 점을 이어 닫힌 도형(사각형, 별),
`false`면 열린 선(스프링, 안테나). `pts[(i + 1) % n]`의 `% n`이
"마지막 다음은 처음"을 만들어 주는 순환 인덱스입니다.

## 5. `wobblyBlob` — 삐뚤빼뚤한 원 (그리고 중점 기법 ⭐)

주인공 몸통, 달, 입에 쓰이는 함수. 두 단계입니다.

**1단계: 반지름을 흔들면서 원 둘레에 점 12개 찍기**

```ts
for (let i = 0; i < 12; i++) {
  const a = (i / 12) * Math.PI * 2;          // 360도를 12등분
  const rr = r + wob(seed, i) * amp;          // 반지름이 점마다 다름!
  pts.push([x + Math.cos(a) * rr, y + Math.sin(a) * rr]);
}
```

완벽한 원이 아니라 감자 같은 몽글몽글한 형태가 됩니다.

**2단계: 그 점들을 "중점 기법"으로 부드럽게 잇기**

```ts
let m = mid(pts[11], pts[0]);         // 이웃한 두 점의 중간에서 출발
ctx.moveTo(m[0], m[1]);
for (let i = 0; i < 12; i++) {
  m = mid(pts[i], pts[(i + 1) % 12]);
  ctx.quadraticCurveTo(pts[i][0], pts[i][1], m[0], m[1]);
  //                   └ 제어점 = 꼭짓점 ┘   └ 끝점 = 다음 중점 ┘
}
```

이게 **캔버스 부드러운 곡선의 정석 관용구**입니다. 원리:

```
   P0 ─────── P1 ─────── P2        P = 꼭짓점,  M = 이웃 점끼리의 중점
       ╲ M0  ╱   ╲  M1 ╱
        ╲__╱      ╲__╱          곡선은 M0 → (P1 쪽으로 당겨졌다가) → M1
```

**왜 꼭짓점이 아니라 중점을 지나가게 할까?**
`quadraticCurveTo`의 곡선은 제어점을 통과하지 않고 스쳐 지나갑니다.
만약 곡선의 끝점을 꼭짓점으로 잡으면 그 지점에서 방향이 꺾여 **모서리**가
생겨요. 끝점을 중점으로 잡으면 이전 곡선이 끝나는 방향과 다음 곡선이
시작하는 방향이 자동으로 일치해서, 전체가 모서리 없이 매끄럽게 이어집니다.

📌 이 기법은 손글씨 앱, 서명 패드, 그림판 등 **부드러운 자유곡선이 필요한
모든 곳**에 쓰입니다. `lineTo`로 이으면 각지고, 3차 베지어는 제어점 계산이
복잡하고 — 중점 기법이 딱 좋은 균형점이에요.

## 6. 도형 재료 함수들 — 점 찍기 3형제

`doodle.ts`의 나머지는 "꼭짓점 배열을 만들어주는" 순수 함수들입니다:

```ts
rectPts(x, y, w, h)      // 사각형 4점 — 위성 본체, 캔
ngonPts(n, r, rot)       // 정n각형 — 볼트(6각형)
spikyPts(n, rIn, rOut)   // 뾰족뾰족 — 별(5), 가시(9)
```

`spikyPts`의 아이디어가 재밌습니다. **바깥 반지름과 안쪽 반지름을 번갈아**
찍으면 뾰족해져요:

```
     rOut ●
         ╱ ╲        짝수 번째 점 → 바깥 원 위에
    rIn ●   ● rIn   홀수 번째 점 → 안쪽 원 위에
       ╱     ╲      직선으로 이으면 = ★
```

이 점들을 `wobblyPath`에 넣으면 "손으로 그린 별"이 완성됩니다.
**"점 만들기"와 "선 흔들기"를 분리**했기 때문에 어떤 도형이든 재료만
갈아끼우면 손그림이 되는 구조예요. 새 쓰레기 종류를 만들 때도 이 3형제로
점만 찍으면 됩니다.

## 7. `fillStroke` — 색연필 채색법

```ts
const fillStroke = (color: string, fillAlpha = 0.15) => {
  const ga = ctx.globalAlpha;
  ctx.globalAlpha = ga * fillAlpha;   // ① 기존 투명도에 "곱해서"
  ctx.fill();                          //    속을 연하게 칠하고
  ctx.globalAlpha = ga;                // ② 되돌린 다음
  ctx.stroke();                        //    같은 색으로 진한 외곽선
};
```

**같은 색으로 "속은 연하게 + 선은 진하게"** — 색연필/마커로 칠한 느낌의
공식입니다. 채도 높은 단색으로 꽉 채우면 벡터 아이콘처럼 딱딱해져요.

⚠️ **①에서 곱셈(`ga * fillAlpha`)이 중요합니다.** `ctx.globalAlpha = fillAlpha`로
덮어쓰면, 무적 깜빡임처럼 바깥에서 걸어둔 투명도(0.4)가 무시되어
"깜빡이는 중인데 몸통 속만 안 깜빡이는" 버그가 납니다.
**바깥의 상태를 존중하려면 곱하고, 무시하려면 대입** — 항상 곱셈이 안전해요.

> 💡 실제로 있었던 튜닝: 주인공 몸통을 기본값 0.15로 칠했더니 어두운 우주
> 배경에서 속이 거의 안 보여 유령 같았습니다. 그래서 `mascot.ts`에서만
> `fillStroke(MASCOT.body, 0.34)`로 올렸어요. **배경색과 채움 투명도는
> 항상 함께 튜닝**해야 합니다.

## 8. 전체 조립 — 위성 하나가 그려지기까지

`debris.ts`의 satellite 케이스를 따라가 봅시다:

```
drawDebris()                        ← 좌표계를 물체 위치로 이동+회전+축소
 └─ drawJunkShape(kind="satellite")
     ├─ [-1, +1] 루프               ← 좌우 태양전지판 (대칭은 루프로!)
     │   ├─ rectPts(...)            ← 패널 사각형 점 4개
     │   ├─ wobblyPath(seed + dir)  ← 손그림으로 흔들기 (좌우 다른 시드!)
     │   └─ fillStroke("#ffd166")   ← 노랗게 색연필 칠
     ├─ 본체 rectPts + wobblyPath(seed + 7) + fillStroke
     ├─ 안테나 wobblyLine + 끝 동그라미 arc
     └─ 눈 2개(arc) + 웃는 입(arc)   ← 위성도 귀엽게
```

여기서 배울 점 두 가지:

1. **모든 치수가 `s`(크기)의 배수** — `s * 0.85`, `s * 1.1`…
   하드코딩된 픽셀이 없어서 `size`만 바꾸면 비율이 유지된 채 커지고 작아집니다.
2. **부품마다 시드를 어긋나게** (`seed + dir`, `seed + 7`) — 같은 시드면
   왼쪽 패널과 오른쪽 패널이 똑같은 방향으로 삐뚤어져 부자연스럽거든요.

## 9. 정리 — 손그림 시스템의 설계도

```
constants.ts     "무슨 색으로"
doodle.ts        "어떻게 흔들어서" (wob, phase, wobblyPath/Blob, fillStroke)
debris/mascot/   "무엇을"         (점 찍기 + doodle 도구로 그리기)
backdrop.ts
joops-game.tsx   "언제"           (draw()에서 setPhase로 장면 넘기기)
```

이 분리 덕분에: 새 캐릭터를 추가할 때는 `doodle.ts`를 건드릴 필요 없이
점만 찍으면 되고, 끓는 속도를 바꿀 때는 `setPhase` 한 줄만 만지면 됩니다.
**각 파일이 딱 하나의 질문에만 답하는 구조** — 좋은 설계란 이런 거예요.
